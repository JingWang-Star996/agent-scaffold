# 需求翻译官（脚手架）— 后置检查模式

> **定位**：这是 LLM Agent 养成过程中的脚手架/辅助轮。
> - 新手期：装上，帮助建立"执行后检查需求符合度"的习惯
> - 成熟期：内化后拆掉，自动检测需求偏离
> - 可随时启用/禁用，不是强制规则

任务执行后，自动检查执行结果是否符合用户真实需求。从用户的抱怨、修正、妥协信号中推断需求偏离，并提出修正建议。

## 何时使用

**启用场景**：
- 任务执行完成后
- 用户说"不是这样的"（否定信号）
- 用户说"算了就这样吧"（妥协信号，通常意味着不满意）
- 用户说"能不能再改改"（迭代信号）
- 想要建立"执行后检查需求符合度"的习惯

**禁用场景**：
- 简单明确的请求（"帮我写个 Hello World"）
- 用户明确说"不用检查"
- 低成本任务，迭代便宜
- 已经内化了需求理解能力

**触发条件**（启用后）：
- 任务执行完成
- 检测到用户反馈中的隐式信号（否定、修正、妥协、迭代、沮丧）
- 执行结果与原始需求存在偏差

## 核心原则

**4 条简单规则**：

1. **执行后检查** — 不要假设执行结果符合需求，验证
2. **从信号中推断** — 用户的抱怨、修正、妥协都是需求信号
3. **不要过度解读** — 只有检测到明确信号时才介入
4. **提供修正选项** — 发现偏离时，给用户提供修正方向

## 工作流程

### 阶段 1：任务执行
```
用户：帮我做个网站
Agent：（直接执行，不做前置澄清）
Agent：[生成网站代码]
```

### 阶段 2：信号检测
```
用户：不是这样的，我想要更简单的

翻译官：（检测到否定信号）
- 信号类型：否定
- 信号强度：高
- 推断：执行结果偏离用户真实需求
```

### 阶段 3：需求推断
```
翻译官：（分析用户反馈）
用户说"更简单的" → 推断真实需求：
- 当前实现：复杂的多页面网站
- 用户期望：简单的单页面网站
- 偏离维度：复杂度
```

### 阶段 4：修正建议
```
翻译官：我注意到你说"更简单的"，我理解你可能想要：
A. 单页面网站（所有信息在一页）
B. 减少功能（只保留核心功能）
C. 简化设计（更少的视觉元素）
D. 其他（请说明）

你想往哪个方向调整？
```

### 阶段 5：重新执行
```
用户：A

翻译官：明白了，需求已更新：
- 类型：单页面网站
- 风格：极简
- 核心内容：[从原需求提取]

确认这个理解对吗？

用户：对

翻译官：（将更新后的需求传递给执行 AI，重新执行）
```

## 信号检测表

| 信号 | 含义 | 行动 |
|------|------|------|
| "不是这样的" | 否定 | 推断偏离维度，提供修正选项 |
| "我是说..." | 修正 | 提取修正内容，更新需求 |
| "算了就这样吧" | 妥协 | 标记不满意，提供重新开始选项 |
| "能不能再改改" | 迭代 | 询问具体迭代方向 |
| "烦了" | 沮丧 | 道歉，提供从头开始选项 |
| "差不多" | 部分满意 | 询问哪些地方需要改进 |

## 反模式

❌ **前置澄清**：执行前问一堆问题（用户烦，打断流程）

❌ **假设理解**：执行前假设自己理解用户需求（危险，导致返工）

❌ **忽视信号**：用户说"不是这样的"但继续执行（无视反馈）

❌ **过度解读**：用户说"还行"就认为完全满意（可能只是客气）

✅ **后置检查**：执行后检查需求符合度（不打断流程，及时纠偏）

✅ **信号驱动**：只有检测到明确信号时才介入（避免过度干预）

✅ **提供选项**：发现偏离时给用户提供修正方向（降低认知负荷）

## 示例对话

```
用户：帮我写篇文章

Agent：[直接执行，生成文章]

用户：不是这种风格，太学术了

翻译官：（检测到否定信号 + 风格偏离）
我注意到你说"太学术了"，你可能想要：
A. 更口语化（像聊天一样）
B. 更商业化（像营销文案）
C. 更通俗化（像科普文章）
D. 其他（请说明）

你想往哪个方向调整？

用户：A

翻译官：明白了，需求已更新：
- 类型：文章
- 风格：口语化（像聊天一样）
- 其他需求：[从原需求提取]

确认这个理解对吗？

用户：对

翻译官：（将更新后的需求传递给执行 AI，重新执行）
```

## 与执行 AI 的集成

检测到需求偏离后，将更新后的需求传递给执行 AI：
```
你是一个执行者。根据更新后的需求重新完成任务。

【原始需求】
- 类型：文章
- 主题：AI Agent
- 长度：2000字

【需求更新】
- 风格：从"学术化"改为"口语化"（像聊天一样）

【任务】
重新写这篇文章，保持主题和长度不变，但风格改为口语化。
```

## 成功指标

| 指标 | 目标 |
|------|------|
| 需求偏离检测率 | > 80%（用户反馈"不是这样的"时能检测到） |
| 修正建议准确率 | > 70%（用户接受修正建议的比例） |
| 二次执行满意度 | > 4.0/5.0 |
| 检查开销 | < 3 秒（不要拖慢速度） |

## 哲学基础

这个脚手架的设计遵循"化繁就简"原则：

**问题**：用户与 AI 直接对话时，Agent 往往假设自己理解了用户需求，直接执行。但执行结果往往偏离用户真实需求，导致返工。

**传统方案**：前置澄清（执行前问一堆问题）— 用户烦，打断流程

**脚手架方案**：后置检查（执行后检测需求偏离）— 不打断流程，及时纠偏

**核心洞察**：
- 用户往往不知道自己想要什么，直到看到不想要的东西
- 用户的抱怨、修正、妥协都是需求信号
- 从信号中推断真实需求，比前置澄清更有效

**理论基础**：
- 认知负荷理论（Sweller, 1988）— 减少用户认知负担
- 需求工程（Davis, 1993）— 用户往往不知道自己需要什么
- 服务恢复悖论（Zeithaml, 1990）— 良好的服务恢复比无问题更满意

## 何时拆除脚手架

当以下条件满足时，可以禁用这个脚手架：
- ✅ 已经能准确理解用户需求，执行结果很少偏离
- ✅ 不需要后置检查就能一次性做对
- ✅ 用户反馈"你越来越懂我了"
- ✅ 首次执行满意度稳定 > 80%

拆除方法：
```bash
# Hermes
rm -rf ~/.hermes/skills/requirement-mediator

# OpenClaw
rm -rf ~/.openclaw/skills/requirement-mediator
```
